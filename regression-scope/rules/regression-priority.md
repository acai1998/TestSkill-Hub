# 回归优先级判断规则

## 优先级分类

### 🔴 必须回归
满足以下任一条件：
- 直接修改的功能模块
- 核心主流程（下单、支付、登录、注册等）受影响
- 修复的 P0/P1 Bug 涉及的功能
- 权限/鉴权逻辑发生变化
- 数据库 Schema 或核心数据结构改变

**测试方式**：完整功能回归，不能只跑冒烟

### 🟡 推荐回归
满足以下任一条件：
- 间接依赖变更点的功能
- 与变更点共享数据的功能
- 修复的 P2 Bug 涉及的功能
- 公共组件修改影响的页面

**测试方式**：重点场景回归，重点验证主流程

### 🟢 可选回归
满足以下条件：
- 与变更点耦合极低
- 纯前端样式/文案修改，逻辑无变化
- 只影响低频边缘场景
- 本次版本未改动且无依赖关系

**测试方式**：冒烟验证即可，或根据时间排期决定

## 高风险场景（强制扩大范围）
以下场景无论变更点范围多小，都必须扩大回归范围：

| 场景 | 扩大范围 |
| --- | --- |
| 支付相关改动 | 所有支付方式 + 订单状态流转全流程 |
| 账号/登录改动 | 所有登录方式 + 权限验证 + SSO 场景 |
| 数据库迁移 | 所有使用迁移表的功能 + 历史数据兼容性 |
| 底层框架升级 | 所有主流程冒烟 |
| 安全修复 | 所有涉及数据安全和权限的模块 |

## 范围缩减原则
以下情况可以减少回归范围，但必须在报告中说明：

| 情况 | 可缩减范围 | 必须说明 |
| --- | --- | --- |
| 纯 UI 改动（无逻辑） | 跳过接口测试 | 确认无逻辑变更 |
| 隔离性强的新功能 | 跳过无依赖的历史模块 | 确认无公共依赖 |
| 配置项关闭的功能 | 跳过该功能的详细用例 | 确认开关处于关闭状态 |

## 注意事项
1. 优先级评估以影响用户量和业务风险为核心考量。
2. 时间紧张时，优先保证"必须回归"项全覆盖。
3. "可选回归"不等于"不用回归"，有时间应尽量覆盖。
