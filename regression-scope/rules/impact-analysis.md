# 影响范围分析方法

## 分析框架：三层影响圈

```
┌─────────────────────────────────┐
│     间接影响（上下游依赖）        │
│  ┌───────────────────────────┐  │
│  │    直接影响（功能本身）     │  │
│  │  ┌─────────────────────┐  │  │
│  │  │   变更点（核心改动） │  │  │
│  │  └─────────────────────┘  │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

## 第一层：变更点分析
识别本次改动的具体内容：
- 哪些接口/函数/类被修改
- 哪些数据字段发生变化
- 哪些业务规则调整
- 哪些配置或环境依赖改变

## 第二层：直接影响
直接调用或直接依赖变更点的功能：

### 前端改动
- 改动的页面/组件：所有使用该组件的页面
- 修改的状态管理：所有读取/写入该状态的模块
- 修改的路由：所有跳转到该路由的入口

### 后端改动
- 修改的接口：所有调用该接口的前端页面/客户端
- 修改的数据库字段：所有读写该字段的接口
- 修改的中间件/拦截器：所有经过该中间件的请求

### 公共组件/库改动
- 所有引用该组件的模块
- 所有依赖该库的功能

## 第三层：间接影响（上下游依赖）

### 上游依赖
哪些功能会触发变更点：
- 例：修改了"下单接口"，上游是"购物车结算"、"立即购买"

### 下游影响
变更点的输出会影响哪些功能：
- 例：修改了"订单状态"字段，下游是"订单列表"、"订单详情"、"退款流程"

### 数据流影响
- 修改数据结构后，影响的报表/统计功能
- 修改缓存策略后，影响的查询接口

## 特殊场景分析

### 权限/鉴权改动
影响范围：所有需要权限验证的功能
重点：不同角色（用户/商家/管理员）的访问控制

### 配置/开关改动
影响范围：所有读取该配置的模块
重点：默认值变化、灰度场景

### 定时任务/异步任务改动
影响范围：依赖任务结果的业务流程
重点：任务失败/重试/超时场景

### 第三方依赖升级
影响范围：所有使用该依赖的模块
重点：API 兼容性、行为变化

## 分析输出要求
1. 每个影响项必须说明为什么受影响（依赖关系）
2. 区分直接影响和间接影响，不能混在一起
3. 给出推荐的测试方式（全量/冒烟/定向）
