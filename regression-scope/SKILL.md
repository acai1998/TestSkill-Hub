# 回归范围分析器

## 技能介绍
这个技能用于在版本发布或 Bug 修复后，根据变更内容快速分析受影响范围，推荐合理的回归测试策略，避免漏测，同时减少不必要的全量回归。

核心目标：
- 影响分析准确（从变更点出发，识别直接和间接影响）
- 范围推荐合理（区分必测/推荐/可选，不做无效的全量回归）
- 结果可落地（必须产出文件，而不是仅页面展示）

## 何时应用
- 用户描述了需求变更或代码改动，要求"分析回归范围"。
- 用户要求"制定回归测试策略"或"确认这次要回归哪些模块"。
- 用户提供 Bug 修复清单，要求"推荐回归测试点"。
- 版本发布前需要快速圈定回归范围。

## 必须先读的规则
执行本技能时，先阅读：
- `rules/_sections.md`

再根据任务读取对应规则：
- 产物输出：`rules/artifact-file-output.md`
- 影响分析方法：`rules/impact-analysis.md`
- 回归优先级：`rules/regression-priority.md`

## 产出目标（强制）
1. 不能只在对话页面输出结果，必须落地为文件。
2. 默认至少产出 1 份 Markdown 文档：
   - `outputs/analysis/<slug>.md`
3. 最终回复中必须明确给出产物路径。

## 输入模式
- 变更描述模式：用户描述本次改动的功能点或代码模块。
- Bug 修复模式：用户提供 Bug 修复列表，分析回归范围。
- 混合模式：新功能 + Bug 修复同时存在。
- 代码 diff 模式：用户提供代码改动摘要，AI 分析影响。

## 执行原则
1. 从变更点出发，识别直接影响（改动功能本身）和间接影响（依赖该功能的其他模块）。
2. 区分三个层次：**必须回归**（直接受影响）/ **推荐回归**（间接依赖）/ **可选回归**（低耦合相关）。
3. 对于高风险改动（核心链路、支付、鉴权），适当扩大范围。
4. 信息不完整时，基于常见系统架构合理推断，并在"待确认项"中注明。
5. 输出结果要可直接用于指导测试执行，不能只说"回归相关模块"。

## 标准工作流
复制以下清单并逐步完成：

```text
Task Progress:
- [ ] Step 1: 解析变更内容，识别改动点
- [ ] Step 2: 分析直接影响范围
- [ ] Step 3: 分析间接影响范围（上下游依赖）
- [ ] Step 4: 评估风险等级，确定回归优先级
- [ ] Step 5: 输出回归测试矩阵
- [ ] Step 6: 写入 Markdown 产物
```

### Step 1: 解析变更内容
- 提取：变更的功能模块、接口、数据结构、业务规则。
- 区分变更类型：新增功能 / 功能修改 / Bug 修复 / 重构优化。

### Step 2: 直接影响分析
- 变更功能本身的所有测试点。
- 改动接口的所有调用方。
- 修改的数据字段涉及的所有页面/功能。

### Step 3: 间接影响分析
依据 `rules/impact-analysis.md` 分析：
- 上游依赖：哪些功能会触发变更点
- 下游影响：变更点的输出会影响哪些功能
- 共享组件：修改的公共组件影响的所有页面

### Step 4: 风险评估与优先级
依据 `rules/regression-priority.md` 对每个回归项打优先级标签。

### Step 5: 输出回归矩阵
按"输出模板（默认）"整理完整回归矩阵。

### Step 6: 写入 Markdown 产物
- 目标路径：`outputs/analysis/<slug>.md`
- 文件内容使用"输出模板（默认）"。
- 对话中仅做摘要，并给出文件路径。

## 输出模板（默认）
写入 Markdown 文件时使用以下结构：

```markdown
## 回归范围分析报告

### 一、变更摘要
| 变更类型 | 变更内容 | 涉及模块 |
| --- | --- | --- |
| 新增功能 | ... | ... |
| 功能修改 | ... | ... |
| Bug 修复 | ... | ... |

### 二、影响范围分析

#### 直接影响
| 模块 | 功能点 | 影响说明 |
| --- | --- | --- |

#### 间接影响
| 模块 | 功能点 | 影响说明 | 依赖关系 |
| --- | --- | --- | --- |

### 三、回归测试矩阵
| 优先级 | 模块 | 回归测试点 | 回归原因 | 建议测试方式 |
| --- | --- | --- | --- | --- |
| 🔴 必须回归 | | | | |
| 🟡 推荐回归 | | | | |
| 🟢 可选回归 | | | | |

### 四、回归策略建议
**整体策略：**
（全量/定向/冒烟 + 理由）

**测试重点：**
- 重点 1：...
- 重点 2：...

**可跳过范围：**
- ...（原因：...）

### 五、待确认项
- [ ] ...
```

## 质量门槛
- 每个回归项必须说明回归原因（为什么要回归）。
- 必须回归项不能遗漏核心主流程。
- 可跳过范围必须给出理由，不能随意缩减。
- 结果必须落地到文件路径，不能只停留在会话中。

## 快速触发示例
- "这次改了用户登录模块的 Token 刷新逻辑，帮我分析回归范围"
- "修复了 3 个 Bug：<Bug列表>，帮我推荐回归测试点"
- "本次版本变更如下：<变更描述>，帮我出回归测试矩阵并落地到文件"
