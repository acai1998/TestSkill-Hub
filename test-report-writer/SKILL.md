# 测试报告生成器

## 技能介绍
这个技能用于将测试执行结果、Bug 数据、遗留问题等信息，快速转为规范、完整的测试报告，供团队评审和上线决策使用。

核心目标：
- 数据准确（通过率、Bug 统计、覆盖率清晰呈现）
- 结论明确（质量评估、上线建议一目了然）
- 结果可落地（必须产出文件，而不是仅页面展示）

## 何时应用
- 用户提供测试结果数据，要求"生成测试报告"。
- 用户要求"输出版本质量报告"或"上线前测试总结"。
- 用户要求对测试结果做质量评估和上线建议。
- 版本发布前需要对外汇报测试情况。

## 必须先读的规则
执行本技能时，先阅读：
- `rules/_sections.md`

再根据任务读取对应规则：
- 产物输出：`rules/artifact-file-output.md`
- 质量评估：`rules/quality-assessment.md`
- 上线建议：`rules/release-recommendation.md`

## 产出目标（强制）
1. 不能只在对话页面输出结果，必须落地为文件。
2. 默认至少产出 1 份 Markdown 文档：
   - `outputs/reports/<slug>.md`
3. 最终回复中必须明确给出产物路径。

## 输入模式
- 数据模式：用户提供用例数、通过数、Bug 列表等结构化数据。
- 描述模式：用户用自然语言描述测试情况，AI 整理成报告。
- 混合模式：部分数据 + 部分文字描述。

## 执行原则
1. 优先从用户提供的数据中计算通过率、Bug 密度等指标。
2. 数据不完整时，基于已有信息合理推断，并在"待确认项"中列出。
3. 质量结论必须有数据支撑，不能只说"测试通过"。
4. 上线建议必须明确：建议上线 / 条件上线（需修复 xxx）/ 建议不上线。
5. 遗留 Bug 必须逐条列出严重等级，不能笼统描述。

## 标准工作流
复制以下清单并逐步完成：

```text
Task Progress:
- [ ] Step 1: 收集并整理测试输入数据
- [ ] Step 2: 计算核心质量指标
- [ ] Step 3: 评估质量等级
- [ ] Step 4: 分析遗留风险，给出上线建议
- [ ] Step 5: 生成报告内容
- [ ] Step 6: 写入 Markdown 产物
```

### Step 1: 收集输入数据
- 必要数据：版本号、测试范围、用例总数、通过数、失败数、Block 数。
- Bug 数据：按严重等级分类统计（P0/P1/P2/P3），已修复数、遗留数。
- 若数据缺失，在"待确认项"中列出，并基于已有数据继续。

### Step 2: 计算质量指标
- 用例通过率 = 通过数 / 执行总数 × 100%
- Bug 修复率 = 已修复数 / 发现总数 × 100%
- 缺陷密度 = Bug 总数 / 测试用例数（参考值）
- 遗留 P0/P1 数量（关键指标）

### Step 3: 质量等级评估
依据 `rules/quality-assessment.md` 中的评级标准。

### Step 4: 上线建议
依据 `rules/release-recommendation.md` 给出明确建议。

### Step 5: 生成报告内容
按"输出模板（默认）"填写完整内容。

### Step 6: 写入 Markdown 产物
- 目标路径：`outputs/reports/<slug>.md`
- 文件内容使用"输出模板（默认）"。
- 对话中仅做摘要，并给出文件路径。

```markdown

| 测试人员 | - |
| 报告日期 | YYYY-MM-DD |

### 二、测试范围
**测试覆盖：**
- 功能模块 1
- 功能模块 2

**不测范围：**
- ...

### 三、测试结果统计

#### 用例执行情况
| 指标 | 数值 |
| --- | --- |
| 计划用例数 | - |
| 实际执行数 | - |
| 通过数 | - |
| 失败数 | - |
| Block 数 | - |
| 用例通过率 | -% |

#### 缺陷统计
| 严重等级 | 发现数 | 已修复 | 遗留 |
| --- | --- | --- | --- |
| P0（致命） | 0 | 0 | 0 |
| P1（严重） | 0 | 0 | 0 |

| P3（轻微） | 0 | 0 | 0 |

Bug 修复率：--%

### 四、遗留问题清单
| Bug ID | 标题 | 严重等级 | 遗留原因 | 处理计划 |
| --- | --- | --- | --- | --- |

**质量等级：** 优秀 / 良好 / 一般 / 较差

（基于通过率、Bug 分布、遗留风险的综合评估）

**主要风险：**
- 风险 1：...
- 风险 2：...

### 六、上线建议
**建议：** ✅ 建议上线 / ⚠️ 条件上线 / ❌ 建议不上线

**上线前必须完成：**
- [ ] ...

- [ ] ...
```

## 质量门槛
- 必须包含量化数据，不能只有文字描述。
- 遗留 Bug 必须逐条列出，不能只写"有 x 个遗留"。
- 上线建议必须明确，不能模糊表达。
- 结果必须落地到文件路径，不能只停留在会话中。

## 快速触发示例
- "帮我生成测试报告，执行了 200 条用例，通过 185 条，发现 15 个 Bug，遗留 3 个 P2"
- "这是本次版本的测试情况：<描述>，帮我整理成测试报告"
- "版本要上线了，帮我出一份测试总结报告并落地到文件"
