# 严重等级与优先级定义

## 严重等级（Severity）

| 等级 | 标识 | 定义 | 典型示例 |
| --- | --- | --- | --- |
| 致命 | P0 | 核心功能完全不可用、数据丢失/损坏、安全漏洞、系统崩溃 | 支付失败、用户数据泄露、App 闪退 |
| 严重 | P1 | 核心功能异常，有 Workaround 但严重影响使用 | 下单后状态不更新、登录偶发失败 |
| 一般 | P2 | 非核心功能异常，UI/体验问题影响较大，或边界场景错误 | 筛选条件不生效、Toast 提示内容错误 |
| 轻微 | P3 | 细微 UI 问题、文案/图标错误、低频场景小问题 | 按钮颜色偏差、文案错别字 |

## 优先级（Priority）

| 优先级 | 处理时机 | 判断依据 |
| --- | --- | --- |
| 高 | 当前版本必须修复 | P0/P1 级别，或影响发布的 P2 |
| 中 | 下一版本修复 | P2 级别，不影响当前发布 |
| 低 | 有时间再处理 | P3 级别，或极低频场景 |

## 判断原则
1. 严重等级由缺陷影响范围和严重程度决定，与发布计划无关。
2. 优先级综合考虑：严重等级 + 影响用户量 + 发布节点。
3. 同一严重等级可以有不同优先级（如 P2 级别的发布阻塞问题优先级为高）。
