# Bug 单生成器

## 技能介绍
这个技能用于将测试人员描述的缺陷现象、复现过程、截图说明等信息，快速转为规范、完整、可直接提交的 Bug 单。

核心目标：
- 信息完整（标题、复现步骤、预期/实际结果、环境、严重等级一应俱全）
- 格式规范（便于开发快速定位、便于测试跟踪关闭）
- 结果可落地（必须产出文件，而不是仅页面展示）

## 何时应用
- 用户描述了一个 Bug 现象，要求"帮我写成 Bug 单"。
- 用户要求"整理缺陷报告"或"规范化这条 Bug"。
- 用户提供截图/日志/复现步骤，要求生成标准格式。
- 用户要求批量整理多个 Bug。

## 必须先读的规则
执行本技能时，先阅读：
- `rules/_sections.md`

再根据任务读取对应规则：
- 产物输出：`rules/artifact-file-output.md`
- 严重等级定义：`rules/severity-priority.md`
- 字段规范：`rules/field-standards.md`

## 产出目标（强制）
1. 不能只在对话页面输出结果，必须落地为文件。
2. 默认至少产出 1 份 Markdown 文档：
   - `outputs/reports/<slug>.md`
3. 多个 Bug 时，可合并到同一文件，按 `## Bug-001`、`## Bug-002` 依次编号区分，文件名使用批次描述（如 `login-bugs-20260323.md`）。
4. 最终回复中必须明确给出产物路径。

## 输入模式
- 简述模式：用户简单描述现象，AI 补全所有字段。
- 详细模式：用户提供完整复现步骤、日志、截图说明。
- 批量模式：用户提供多个 Bug 描述，批量生成。

## 执行原则
1. 先理解缺陷本质（是功能错误、性能问题、UI 问题还是数据问题）。
2. 信息不完整时，基于合理推断补全，并在"待确认项"中列出。
3. 标题必须简洁准确，格式：`[模块] 现象描述（触发条件）`。
4. 复现步骤必须可独立执行，不依赖上下文隐含知识。
5. 严重等级和优先级必须依据规则给出，不能随意填写。

## 标准工作流
复制以下清单并逐步完成：

```text
Task Progress:
- [ ] Step 1: 理解缺陷现象与影响范围
- [ ] Step 2: 补全缺失信息（环境、版本、账号等）
- [ ] Step 3: 确定严重等级与优先级
- [ ] Step 4: 生成规范 Bug 单内容
- [ ] Step 5: 写入 Markdown 产物
```

### Step 1: 理解缺陷现象
- 提取：触发条件、异常现象、影响用户/功能范围。
- 判断缺陷类型：功能错误 / 性能 / UI/UX / 数据 / 安全 / 兼容性。

### Step 2: 补全缺失信息
- 必要字段：测试环境、App 版本/接口版本、操作系统/浏览器、测试账号类型。
- 若信息缺失，基于常见场景合理推断，并在"待确认项"中注明。

### Step 3: 确定严重等级与优先级
严重等级（Severity）：
- **P0/致命**：核心功能完全不可用、数据丢失、安全漏洞
- **P1/严重**：核心功能异常，有 Workaround 但影响大
- **P2/一般**：非核心功能异常，或 UI/体验问题影响较大
- **P3/轻微**：细微 UI 问题、文案错误、低频场景异常

优先级（Priority）：结合严重等级 + 发布计划 + 影响用户量综合判断。

### Step 4: 生成 Bug 单
按"输出模板（默认）"填写完整字段。

### Step 5: 写入 Markdown 产物
- 目标路径：`outputs/reports/<slug>.md`
- 文件内容使用"输出模板（默认）"。
- 对话中仅做摘要，并给出文件路径。

## 输出模板（默认）
写入 Markdown 文件时使用以下结构：

```markdown
## Bug 单

### 基本信息
| 字段 | 内容 |
| --- | --- |
| 标题 | [模块] 现象描述（触发条件） |
| 严重等级 | P0/P1/P2/P3 |
| 优先级 | 高/中/低 |
| 缺陷类型 | 功能/性能/UI/数据/安全/兼容性 |
| 发现版本 | x.x.x |
| 所属模块 | - |
| 指派给 | - |

### 环境信息
| 字段 | 内容 |
| --- | --- |
| 测试环境 | 预发/测试/线上 |
| 操作系统 | - |
| 客户端版本 | App x.x / 浏览器 Chrome xxx |
| 接口版本 | - |
| 测试账号 | 账号类型/角色 |

### 问题描述
**现象：**
（一句话描述看到了什么）

**影响范围：**
（影响哪些用户、哪些功能）

### 复现步骤
1. 前置条件：
2. 操作步骤：
   - Step 1:
   - Step 2:
   - Step 3:

### 预期结果
（正确情况下应该发生什么）

### 实际结果
（当前实际发生了什么）

### 附件
- 截图/录屏：
- 日志摘要：
- 接口响应：

### 待确认项
- [ ] ...
```

## 质量门槛
- 标题格式：`[模块] 现象（触发条件）`，不超过 50 字。
- 复现步骤必须包含前置条件，每步操作明确。
- 预期结果和实际结果必须形成对比，可观察可验证。
- 严重等级必须给出，不能为空。
- 结果必须落地到文件路径，不能只停留在会话中。

## 快速触发示例
- "帮我把这个 Bug 整理成规范的 Bug 单：<现象描述>"
- "我发现了一个问题：<描述>，帮我写成 Bug 报告。"
- "批量整理以下几个缺陷：<Bug 列表>"
